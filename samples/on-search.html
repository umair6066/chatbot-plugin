onSearch — API-backed search

For large catalogs, pass an onSearch callback instead of loading all products upfront. The widget calls it for every non-greeting message and shows the returned products as cards.

⚠ This demo simulates an API with a local dataset and a 400 ms artificial delay. In production, replace the body of onSearch with a real fetch() call.

Real-world usage

ChatbotWidget.init({
  onSearch: async (query) => {
    const res = await fetch(
      `/admin/products?search=${encodeURIComponent(query)}`
      + `&status=Purchased/InStock,Consignment,Working On`
      + `&per_page=5&page=1`
    );
    const data = await res.json();
    return data.data.products.map(mapProduct); // your mapping fn
  },
  suggestions: [
    "Search Rolex",
    "What's in stock?",
    "Search by ref number",
    "Show me prices",
  ],
});

How it works

onSearch is called for every non-greeting message. Greetings ("hi", "thanks", etc.) are handled locally — no API call is made. Results are capped at 5 cards per reply regardless of how many your API returns. The widget still applies local intent handling (cheapest, prices, stock) on whatever your callback returns, so those intents keep working with API-sourced products.

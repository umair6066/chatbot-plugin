RELEASE NOTES

Chatbot Plugin Update

23 June 2026  ·  All improvements are live

New Features

New

Show More — result pagination

Results are capped at 5 cards per reply to keep the chat readable. When there are more matches a "Show 5 more" chip appears automatically. Tapping it pages through the full result set without re-running the search.

New

Price Range Search

Users can now filter by price in plain language:

"under 10k" "less than $15,000" "between 5k and 20k" "over 50k"

Also works as a refinement — search "rolex" first, then "under 15k" to narrow just those results.

New

Follow-up by Position

After any product listing, users can refer to results by position — no need to type the full product name:

"tell me about the second one" "the third watch" "#2"

The bot remembers what it last showed and picks the right card.

New

Generic Product Types

The Product type is now generic — each client defines their own product shape with full TypeScript safety. Base fields are always available; extra fields are added as the generic parameter: 

type WatchProduct = Product<{
  model:   string;
  reg:     string;
  metal?:  string;
  dial?:   string;
  serial?: string;
  status?: string;
}>;

Custom fields are automatically searched and rendered — no extra configuration. A user searching ref=116610LN or blue dial steel matches against every field including custom ones. Works across all integration methods:

Method Type safety Custom fields searched & rendered
Inline array (TypeScript) ✓ Compile-time ✓
productsUrl JSON fetch Runtime only ✓
IIFE / CDN (ChatbotWidget.init) Plain JS ✓
New

API-Backed Search — onSearch

For large or live catalogs, pass an onSearch callback instead of loading products upfront. The widget calls it for every non-greeting message and renders the returned products as cards: 

<ChatbotWidget
  onSearch={async (query) => {
    const res = await fetch(
      `/api/products/search?q=${encodeURIComponent(query)}`
    );
    const data = await res.json();
    return data.products;
  }}
/>
  • Greetings ("hi", "thanks") handled locally — no API call made
  • Results capped at 5 cards per reply; "Show more" pages without a second call
  • All local intents (cheapest, price range, stock) still work on API results
  • Friendly error message shown if the API call fails

The on-search.html sample demonstrates this with a simulated 400 ms API delay and a live request log.

Improvements

Improved

Conversational Reply Quality

All bot responses rewritten to sound natural rather than mechanical:

Before
"Found 7 matching products (showing 5 of 7):"

"Here are our in-stock products:"

"I'm not sure I understood that."
After
"Found 7 matches — here are the first 5:"

"9 available right now (1 temporarily unavailable):"

"I didn't quite catch that — try a brand name, model, or ref number:"
Improved

Suggestion Chips

  • Chips update after every bot reply with context-aware options
  • Older chips grey out so it's clear which are active
  • "Show X more" chip added automatically when results are capped
  • After a product detail reply, chips suggest related products from the same results
Improved

React Native Sample App

  • Native tab updated to use the same 10 watch products as the web widget
  • Native search now supports: brand search, ref number, price range, stock, cheapest, most expensive
  • Suggestion chips update after every reply — not just the welcome message
  • WebView tab uses Date.now() as cache buster — always loads the latest widget on app start, no manual version bumping needed
